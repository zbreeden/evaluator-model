TL;DR

Ship a modular evaluator with three pillars: Model Performance, Data Quality, and Decision Criteria. Track usage with standardized events (evaluator_action) and showcase results in Power BI.

What’s inside

  • Modules: self-contained feature folders under modules/.
  • Apps: apps/ux_playground to emit events and demo flows.
  • Telemetry: GA4 via GTM, no PII. Canonical taxonomy in telemetry/events.json.
  • Artifacts: screenshots/GIFs/exports for PRs and docs.
  • BI: Power BI notes/exports in powerbi/.
  • Governance: Conventional commits, lightweight PR checklist.
  • Local dev: zero-dep static server (scripts/serve.sh).
  • Licensing: MIT.

Quick start

  1. Clone the repo and serve locally.
git clone <your-repo-url> evaluator
cd evaluator
./scripts/serve.sh   # or: python3 -m http.server 5500
  1. Open http://localhost:5500/apps/ux_playground/, click tiles to emit events.
  2. Enable GTM Preview, confirm events in Tag Assistant → GA4 DebugView.
  3. Start a new module by copying the template:
cp -R modules/_template modules/<your-module-slug>

Telemetry: GA4 via GTM

Standardized client events so usage is measurable across modules.

Event taxonomy

{
  "name": "evaluator_action",
  "required_params": ["module", "action"],
  "optional_params": ["context", "ts"],
  "examples": [
    {"module":"model-performance","action":"view_kpi","context":{"kpi":"roc_auc","value":0.87}},
    {"module":"data-quality","action":"run_check","context":{"check":"freshness_hours","value":3}},
    {"module":"decision-criteria","action":"approve","context":{"policy":"launch_gate"}}
  ]
}

GTM wiring (GA4 event)

  1. Create GA4 Configuration tag (Measurement ID from GA4).
  2. Create GA4 Event tag listening to evaluator_action.
  3. Map parameters: module, action, context (stringified), ts.
  4. Preview → trigger actions in the playground → verify in DebugView.

Modules

Model Performance

KPIs (ROC/PR, lift), challenger vs. baseline, drift monitors.

Events:view_kpi, compare

Data Quality

Freshness, completeness, validity; issue surfacing & reports.

Events:run_check, open_report

Decision Criteria

Guardrails and thresholds that define release/rollback gates.

Events:apply_threshold, approve

Power BI views

  • Model Performance View: KPI cards, ROC/PR, lift vs. baseline.
  • Data Quality View: freshness (hrs), completeness %, validity issues.
  • Decision View: criteria table, pass/fail, rationale notes.

Export static PNGs/GIFs to artifacts/ and reference them in PRs.

Architecture (high-level)

User → UX (apps/*) → dataLayer → GTM → GA4 → BigQuery (optional) → Power BI
                     └── interacts with modules: model-performance | data-quality | decision-criteria

A Mermaid diagram lives in docs/architecture.md if you want a rendered graphic.
